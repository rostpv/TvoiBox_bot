# Окончательный стек проекта

Этот документ фиксирует окончательный стек технологий для проекта Telegram-бота и будущего mini app.

Документ нужен, чтобы:

- больше не возвращаться к выбору стека на каждом шаге;
- строить MVP сразу на правильной основе;
- не переделывать архитектуру при добавлении mini app;
- сохранить проект простым, но с хорошим запасом на рост.

## 1. Итоговое решение

Для проекта фиксируем такой основной стек:

- `Node.js 24 Active LTS`
- `TypeScript`
- `pnpm workspaces`
- `NestJS`
- `grammY`
- `Next.js (App Router)`
- `PostgreSQL`
- `Prisma`
- `Google Calendar API`
- `Docker Compose`
- `Caddy`

Дополнительно:

- встроенное логирование backend-приложения;
- серверные и прикладные логи в файлы и консоль;
- webhook-модель для Telegram;
- REST API как внутренняя основа под будущий mini app.

## 2. Почему выбран именно этот стек

### `Node.js 24 Active LTS`

Это базовая платформа проекта.

Почему выбираю именно его:

- это production-ориентированный LTS-вариант;
- подходит и для Telegram-бота, и для backend, и для mini app в одной экосистеме;
- не будет разрыва между серверной и фронтенд-частью;
- проще поддерживать один технологический стек, чем два разных.

Почему не `Python` как основной стек:

- для чистого Telegram-бота Python действительно очень удобен;
- но у нас дальше будет mini app;
- если взять Node.js + TypeScript, то backend, bot-логика и будущий frontend будут жить в одной экосистеме;
- это лучше для долгой жизни проекта, чем разносить проект между Python и JavaScript/TypeScript.

## 3. Язык проекта

### `TypeScript`

Это основной язык всей системы.

Почему выбираю:

- строгая типизация уменьшает число ошибок;
- проще поддерживать проект по мере роста;
- удобно делить модели и контракты между backend, ботом и будущим mini app;
- лучше подходит для долгого сопровождения, чем обычный JavaScript.

Почему не `JavaScript`:

- MVP на нем можно сделать быстрее только в совсем примитивном виде;
- но дальше проект будет сложнее сопровождать;
- мы сразу строим основу с прицелом на развитие.

## 4. Формат репозитория

### `pnpm workspaces`

Я рекомендую monorepo-структуру на `pnpm workspaces`.

Почему:

- можно держать в одном репозитории:
  - backend;
  - bot;
  - mini app;
  - общие типы;
  - общие утилиты;
- удобно переиспользовать типы и модели;
- меньше дублирования;
- проще развивать mini app позже.

Почему не отдельные разрозненные репозитории:

- на старте это дает лишнюю сложность;
- придется дублировать контракты и типы;
- тяжелее поддерживать синхронность изменений.

## 5. Backend-стек

### `NestJS`

Это основной backend-фреймворк проекта.

Почему выбираю:

- хорошо подходит для модульной архитектуры;
- удобно разнести проект на блоки:
  - клиенты;
  - слоты;
  - заявки;
  - тренировки;
  - интеграция с Google Calendar;
  - настройки;
  - выгрузки;
- хорошо подходит под будущий рост;
- удобно тестировать;
- удобно внедрять логирование, валидацию, конфигурацию и сервисные слои;
- хорошо сочетается с TypeScript и Prisma.

Почему не `Express` в чистом виде:

- для MVP можно, но архитектура быстрее превращается в хаос;
- нам нужна не только одна точка входа, а системный проект с дальнейшим развитием.

Почему не `Fastify` как основной самостоятельный фреймворк:

- он быстрый, но нам важнее не микропроизводительность, а управляемость проекта;
- при желании NestJS потом можно запускать и через Fastify adapter, если это когда-то понадобится.

## 6. Telegram-бот

### `grammY`

Это фреймворк для Telegram-бота внутри Node.js-экосистемы.

Почему выбираю:

- современный и очень удобный для Telegram;
- хорошо подходит для кнопок, меню, состояний и webhook;
- легко масштабируется;
- удобно стыкуется с TypeScript;
- подходит как для простого MVP, так и для роста логики.

Почему не другие Telegram-фреймворки:

- `telegraf` тоже хороший, но `grammY` выглядит современнее и чище для новой разработки;
- он лучше подходит для аккуратной поэтапной сборки и хорошо документирован.

Как будет использоваться:

- `grammY` отвечает за входящие Telegram-апдейты, меню, кнопки, пользовательские сценарии и админские сценарии;
- бизнес-логика при этом не должна жить хаотично внутри обработчиков бота, а должна выноситься в backend-сервисы.

## 7. Mini App

### `Next.js (App Router)`

Это стек для будущего mini app.

Почему выбираю:

- это зрелый и современный React-фреймворк;
- удобно развивать mini app как полноценный продукт;
- хорошо сочетается с TypeScript;
- подходит для роста проекта в сторону более сложного интерфейса;
- позволяет позже без боли расширять систему;
- хорошо подходит для общего fullstack-подхода рядом с NestJS.

Почему не `чистый React + Vite` как основной финальный выбор:

- для очень маленького mini app это хороший вариант;
- но с точки зрения долгой жизни системы `Next.js` дает более зрелую основу;
- если mini app вырастет в личный кабинет или что-то большее, Next.js будет удобнее.

Почему не делать mini app сразу:

- это не входит в MVP;
- но стек backend и frontend должен уже сейчас это учитывать.

## 8. Работа с Telegram Mini App

Для mini app мы опираемся на официальный Telegram Web App / Mini App API.

Почему это важно:

- mini app работает как web-интерфейс внутри Telegram;
- ему нужна корректная работа с Telegram-контекстом;
- нужно сразу строить frontend так, чтобы потом он мог безопасно запускаться из Telegram.

Технический принцип:

- mini app будет отдельным frontend-приложением;
- backend будет отдавать ему API;
- Telegram-бот и mini app будут работать с одной и той же бизнес-логикой и одной базой данных.

## 9. База данных

### `PostgreSQL`

Это основная база данных проекта.

Почему выбираю:

- надежная серверная база;
- хорошо подходит для заявок, слотов, статусов, клиентов и истории изменений;
- лучше подходит для реального использования, чем упрощенные варианты;
- легко масштабируется под будущий рост.

Почему не `SQLite`:

- для локального прототипа возможно;
- для реального серверного бота, который будет работать постоянно, это слабее и менее надежно;
- хуже подходит для конкурентных сценариев.

## 10. ORM и миграции

### `Prisma`

Это основной инструмент работы с базой.

Почему выбираю:

- отлично сочетается с TypeScript;
- дает понятную модель данных;
- удобен для миграций;
- проще поддерживать схему базы по мере роста проекта;
- хорошо подходит под monorepo и shared types.

Почему не писать все на “чистом SQL”:

- можно, но будет сложнее сопровождать;
- выше риск ошибок;
- дольше вносить изменения в схему.

Почему не `TypeORM`:

- он рабочий, но для нового TypeScript-проекта `Prisma` обычно более удобен и предсказуем;
- схема данных и миграции в Prisma обычно проще для поддержки.

## 11. Интеграция с календарем

### `Google Calendar API`

Это обязательная интеграция проекта.

Почему выбора почти нет:

- по ТЗ календарь тренера должен быть в Google Calendar;
- подтвержденные тренировки должны туда попадать;
- переносы и отмены должны синхронизироваться.

Какой подход фиксируем:

- бот и backend — главный источник данных;
- Google Calendar — внешний календарь, который синхронизируется из системы;
- ручные изменения в Google Calendar не считаются главным источником правды.

## 12. Контейнеризация и локальный запуск

### `Docker Compose`

Это основной способ локального и серверного запуска.

Почему выбираю:

- одинаковый запуск локально и на сервере;
- проще поднимать приложение и базу вместе;
- меньше проблем “у меня работает, а на сервере нет”;
- легче переносить проект между машинами;
- проще документировать запуск.

Почему не ставить все руками:

- больше вероятность ошибок;
- хуже повторяемость окружения;
- сложнее поддержка.

## 13. Reverse proxy и HTTPS

### `Caddy`

Это внешний веб-сервер перед приложением.

Почему выбираю:

- проще в настройке, чем классический Nginx;
- очень удобен для HTTPS;
- хорошо подходит для webhook;
- снижает ручную инфраструктурную сложность;
- это хороший выбор для проекта, который нужно сделать аккуратно и без лишней возни.

Почему не `Nginx`:

- Nginx отличный и зрелый;
- но для нашего проекта `Caddy` проще в настройке и удобнее для быстрого безопасного старта;
- особенно это хорошо, когда проект ведется маленькой командой и без отдельного DevOps.

## 14. Логирование

### Что фиксируем

- backend-логи;
- bot-логи;
- логи работы с базой в ключевых местах;
- логи синхронизации с Google Calendar;
- логи административных действий;
- логи ошибок webhook;
- логи ручной повторной синхронизации.

Почему это важно:

- мы будем идти по этапам и постоянно проверять результат;
- логи должны помогать быстро понять, что сломалось;
- без логов такой проект будет трудно поддерживать.

## 15. Архитектурный принцип проекта

Фиксируем такой подход:

- один backend на `NestJS`;
- Telegram-бот на `grammY`, но бизнес-логика вынесена в backend-сервисы;
- будущий mini app на `Next.js`;
- единая база `PostgreSQL`;
- единые типы и модели внутри monorepo;
- внешние интеграции идут через отдельные сервисные слои.

Это важно, потому что:

- мы не строим “бот из одного файла”;
- мы строим основу, к которой позже добавится mini app;
- бот и mini app должны пользоваться одной и той же предметной логикой, а не разной реализацией.

## 16. Что не включаем в обязательный стек прямо сейчас

Пока не фиксируем как обязательную часть ядра:

- `Redis`
- очереди задач отдельным брокером
- `Kafka`
- `RabbitMQ`
- отдельный frontend state management enterprise-уровня
- микросервисную архитектуру

Почему:

- для MVP это избыточно;
- усложнит разработку и поддержку;
- не соответствует цели “максимально просто и максимально незатратно”.

Если проект вырастет, эти вещи можно будет добавить позже.

## 17. Окончательный вывод

### Финальный стек проекта

- `Node.js 24 Active LTS`
- `TypeScript`
- `pnpm workspaces`
- `NestJS`
- `grammY`
- `Next.js (App Router)`
- `PostgreSQL`
- `Prisma`
- `Google Calendar API`
- `Docker Compose`
- `Caddy`

### Почему это лучший выбор именно для нашего проекта

- подходит для Telegram-бота уже сейчас;
- подходит для mini app потом;
- позволяет держать один технологический стек;
- не ломает будущий рост проекта;
- дает понятную архитектурную основу;
- не перегружен лишними enterprise-решениями;
- остается реалистичным для MVP;
- удобен для поэтапной разработки вместе с нейросетями.

## 18. Что считаем окончательно зафиксированным

После этого документа считаем зафиксированным:

- backend делаем на `NestJS`;
- bot-часть делаем на `grammY`;
- mini app в будущем делаем на `Next.js`;
- базу берем `PostgreSQL`;
- ORM берем `Prisma`;
- локальный и серверный запуск строим через `Docker Compose`;
- внешний proxy и HTTPS делаем через `Caddy`;
- проект ведем как единый TypeScript monorepo.
