# Web-версия записи: план внедрения

Дата создания: 2026-06-05.

Статус: код web-записи и web-кабинета тренера задеплоен в production; текущий release `bc6a51187be72f10f66e86c2fd17575f41019af9`. После нештатного первого GitHub Actions deploy VPS был перезагружен, включён swap 2 ГБ, вручную пересобраны образы, применена Prisma-схема и подняты контейнеры. Production QA web-flow и Google Calendar прошли; `/booking` и `/trainer` отвечают `200`, бот поднял Telegram webhook. Полный Telegram regression внутри Telegram ещё не выполнялся.

Цель: добавить мобильную web-версию записи вне Telegram, чтобы клиенты могли записываться из обычного браузера, а все заявки синхронизировались с текущей mini app, ботом, тренерским контуром, Google Calendar и общей базой. Сразу предусмотреть web-кабинет тренера как резервный канал на случай, если Telegram недоступен.

## Принятые решения

- Web-запись делаем параллельно текущей Telegram mini app.
- Telegram mini app не ломаем и не заменяем.
- База, API, календарь и тренерский поток остаются общими.
- Авторизация простая, без SMS и без пароля.
- Первый вход: клиент вводит имя и телефон.
- Email необязателен.
- Система запоминает клиента на устройстве.
- Если клиент открыл запись с другого устройства, он может снова ввести телефон, а система подтянет сохранённые данные.
- Тренер должен видеть источник заявки: `Telegram` или `Web`.
- В Google Calendar должны попадать контакты клиента из web-заявки.
- Web-кабинет тренера нужен как резервный контур управления заявками.
- Web-кабинет тренера должен работать с теми же заявками, слотами, календарём и настройками, что и Telegram-бот.
- На первом этапе web-кабинет тренера можно сделать без сложной ролевой системы: один тренер/владелец, защищённый вход.

Результаты обследования:

- Текущий Telegram mini app flow начинается в `apps/mini-app/src/components/mini-app-root.tsx`: session создаётся через Telegram WebApp `initData`, профиль сохраняется через `MiniAppApi.updateProfile`, слоты загружаются через `MiniAppApi.getClientSlots`, заявка отправляется через `MiniAppApi.requestBooking`.
- Frontend API-клиент находится в `apps/mini-app/src/lib/mini-app-api.ts`: клиентские routes идут в `/mini-app/client/*`, тренерские routes идут в `/mini-app/trainer/*`, авторизация везде через Bearer token mini app session.
- API mini app routes находятся в `apps/api/src/modules/mini-app/mini-app.controller.ts`: `POST /mini-app/session`, `GET/POST /mini-app/me`, `GET /mini-app/client/slots`, `POST /mini-app/client/bookings/request`, client trainings/proposals/cancel/reschedule, trainer bookings/trainings/slots/settings/clients/no-slot/export.
- Telegram-пользователь определяется в `apps/api/src/modules/mini-app/mini-app-auth.service.ts`: `createSessionFromInitData` проверяет Telegram `initData`, достаёт `user.id`, создаёт HMAC session token на `MINI_APP_AUTH_SECRET`, роль `trainer` выбирается по `ADMIN_TELEGRAM_ID` или `TRAINER_TELEGRAM_ID`.
- Клиентская модель сейчас Telegram-first: `apps/api/prisma/schema.prisma`, `Client.telegramId String @unique`, `Booking.clientId`, `Slot`, `Training`, `Reschedule`, `NoSlotRequest`, `TrainerSettings`, `CalendarSyncLog`. Для web-клиента нужна миграция: отвязать клиента от обязательного `telegramId`, добавить web/contact identity, session token и source заявки.
- Создание заявки реализовано в `BookingsService.createBookingRequest` (`apps/api/src/modules/bookings/bookings.service.ts`): принимает `telegramId`, проверяет клиента, blacklist, слот, правила записи, переводит слот `OPEN -> HELD`, создаёт `Booking PENDING`, уведомляет тренера через Telegram.
- Слоты и конфликты находятся в `apps/api/src/modules/slots/slots.service.ts` и `BookingsService.resolveSlotForBooking`: есть явные и virtual slots, настройки длительности/рабочих дней/минут, `Slot.status`, уникальный индекс `[startAt, endAt]`, транзакционные проверки `HELD/BOOKED/CLOSED`.
- Подтверждение, отмена и перенос находятся в `BookingsService.confirmBooking`, `cancelConfirmedTraining`, `rescheduleConfirmedTraining`, `cancelTrainingByClient`, `rescheduleTrainingByClient`, `acceptProposedBookingTime`, `declineProposedBookingTime`; их нужно переиспользовать для web-кабинета, чтобы не создавать вторую бизнес-логику.
- Google Calendar создаётся/обновляется в `apps/api/src/modules/google-calendar/google-calendar.service.ts`, вызывается из `BookingsService.confirmBooking` и сценариев переноса; событие содержит имя, телефон, username/Telegram ID и `tvoyBoxTrainingId` в private extended properties. Для Web нужно добавить source и web-контакты.
- `.ics` формируется в `BookingsService.getClientTrainingCalendarFile`, `getTrainerBookingCalendarFile`, `buildIcsContent`; mini app скачивает файл через `/mini-app/client/trainings/calendar` и `/mini-app/trainer/bookings/calendar`.
- Уведомления тренеру и клиенту идут через `apps/api/src/modules/telegram-notifications/telegram-notifications.service.ts`. Для web-клиентов клиентские Telegram-уведомления невозможны без `telegramId`, поэтому потребуется отдельная логика: тренеру Telegram-уведомление оставить, клиенту показывать статус в web и, если будет выбрано, `.ics`/email позже.
- Старый Telegram-бот также использует `apps/bot/src/services/bookings-api-service.ts` и routes `/bookings/*`, где всё передаётся через `telegramId`/`trainerTelegramId`. Эти routes нельзя ломать; новые web endpoints лучше добавить параллельно или вынести общие методы в `BookingsService` по `clientId`.
- Web-страницы лучше добавить в Next.js рядом с текущей mini app: например `apps/mini-app/src/app/booking/page.tsx` и `apps/mini-app/src/app/trainer/page.tsx`, оставив корневую `/` mini app без изменения.
- Нужна миграция базы: минимально сделать `Client.telegramId` nullable/unique, добавить нормализованный телефон как ключ поиска web-клиента, email, source/канал заявки, web session token или отдельную модель web-сессий. Также потребуется расширить DTO, тренерские списки и календарное описание.

Предварительная логика:

- Клиент вводит имя и телефон.
- API нормализует телефон.
- Если клиент с таким телефоном уже есть, API подтягивает профиль.
- Если клиента нет, API создаёт новый профиль.
- Браузер получает session token и сохраняет его.
- Следующие открытия web-записи используют session token.

Принятая схема этапа 1:

- URL web-записи: `https://app.tvoybox.ru/booking`. Это не требует нового DNS и Caddy route, потому что домен mini app уже работает.
- URL web-кабинета тренера: `https://app.tvoybox.ru/trainer`. Отдельный `trainer.tvoybox.ru` оставить на потом, если понадобится вынести кабинет отдельно.
- Первый экран web-записи сразу показывает рабочую форму и слоты; отдельную landing page не делаем. Допускается короткий текст над формой: что это запись на персональную тренировку и тренер подтвердит заявку.
- Обязательные поля нового клиента: имя и телефон. Для повторного входа с другого устройства существующему web-клиенту достаточно телефона. Email и комментарий необязательны.
- Клиент может редактировать сохранённые данные перед отправкой новой заявки.
- Web-клиент видит свои будущие записи, если на устройстве есть web-session token.
- `.ics` для web-клиента показывать ссылкой/кнопкой в web-интерфейсе у подтверждённой тренировки.
- Связь тренера с web-клиентом на первом этапе вручную: телефон обязателен, email используется если указан.
- Web-session token хранить в `localStorage` на клиенте и проверять на сервере по отдельной web-сессии/хешу. Токен должен быть случайным, долгоживущим, с возможностью отзыва через удаление записи сессии.
- Телефон нормализовать на API: оставить только цифры, российские номера приводить к `7XXXXXXXXXX`, для остальных хранить цифровую форму без форматирования. Отображаемый телефон хранить отдельно как ввёл клиент или в мягко отформатированном виде.
- При совпадении нормализованного телефона с существующим клиентом: переиспользовать клиента, обновить имя/email только если пришли непустые значения, Telegram ID не трогать.
- Источник заявки хранить в базе как `Booking.source`: `telegram` для текущих mini app/бот-потоков и `web` для браузерной записи.
- Вход тренера в web-кабинет: простой пароль/секрет из `.env.server`, сервер выдаёт отдельный trainer web-session token. Пароль не хранить в репозитории.
- Действия тренера в первом web-кабинете: видеть заявки и будущие тренировки, источник `Telegram/Web`, контакты, подтверждать, отклонять/отменять, предлагать перенос, менять настройки расписания.

## Риски и ограничения

- Без SMS клиент может ошибиться в телефоне.
- Если клиент очистит браузер, устройство забудет session token.
- Если клиент зайдёт с другого устройства, нужно повторно ввести телефон.
- Нужна аккуратная защита от случайного создания дублей клиента по разному формату телефона.
- Нужна проверка, чтобы web-заявки не обходили ограничения слотов.
- Нельзя ломать Telegram mini app, потому что это текущий рабочий канал.
